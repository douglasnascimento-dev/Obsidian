### 🌿Personas

Criado para ajudar a definir os usuários típicos, uma persona é um personagem fictício, um modelo hipotético que representa o perfil do usuários reais. Personas podem representar diferentes usuários, ou diferentes perfis. As características de uma persona são:
- Identidade → possui um nome e sobrenome, gênero, idade.
- Objetivo → o que ela pretende fazer com o produto? o que ela precisa?
- Habilidades → como é a sua relação com tecnologias?
- Tarefas → quais as tarefas relacionadas ao contexto do produto essa persona realiza?
- Requisitos → o que ela precisa? as expectativas? que tipo de produto agrada essa pessoa?

### 🌿Análise das Tarefas

- Entender, analisar e documentar como as pessoas executam as tarefas com as ferramentas ou sistemas existentes. 
- O Que as pessoas fazem? Quais os artefatos envolvidos? O que elas precisam saber?

- Análise da situação atual → antes da introdução de uma solução
- Análise das tarefas depois → como serão executadas as tarefas com a sua solução

### 🌿Cenários

É a história sobre pessoas realizando uma atividade. A Narrativa é textual ou pictórica de uma situação. Possui um enredo, que inclui uma sequência de ações, de uma execução de uma tarefa em um determinado contexto. Elementos do Cenário:
- Ambiente e contexto, a situação de execução da tarefa;
- Atores, as pessoas envolvidas na tarefa;
- Objetivos, a motivação dos atores;
- Ações, o comportamento observável;
- Eventos, as ações externas produzidas por outras e/ou objetos; e
- Avaliação, a avaliação mental do resultado pelos atores.

### 🌿Níveis de Requisitos

Os requisitos devem ser redigidos de modo a serem passíveis de entendimento pelo diversos interessados, os stakeholders. Clientes, usuários finais e devs são todos interessados em requisitos, mas têm expectativas diferentes. É útil apresentar requisitos em diferentes níveis de descrição. 

- **Requisitos de Usuário** → os requisitos abstratos de alto nível.
- **Requisitos de Sistema** → os requisitos detalhados.

#### 🌱Requisitos de Usuário

São as declarações, em uma **linguagem natural**, de quais serviços são esperados do sistema e as restrições, sob as quais ele deve operar. Servem de base para um entendimento entre clientes e devs acerca do que o sistema deve contemplar. Esses requisitos, normalmente, são usados como base para a contratação e o planejamento do projeto. 

#### 🌱Requisitos de Sistema

Definem, detalhadamente, as funções, os serviços e as restrições operacionais do sistema. O Documentação de requisito de sistema deve ser preciso. Ele **deve definir exatamente** o que será implementado. São versões expandidas dos requisitos de usuário usados como ponto de partida para o projeto do sistemas. 

Os requisitos de sistema são elaborados como parte dos esforços diretos para o desenvolvimento do sistema. São capturados detalhes importantes para as fases técnicas posteriores do processo de desenvolvimento, a saber: projeto, implementação e testes. As especificações em linguagem natural são insuficientes e para especificá-los, notações mais especializadas devem ser utilizadas.

#### 🌱Requisitos Funcionais

Estão diretamente ligados a **comportamento do software**. São declarações de serviços que o sistema deve prover, descrevendo o que sistema deve fazer. Um requisito funcional descreve uma interação entre o sistema e o qualquer ator ou entidade externa que estará interagindo com o sistema. 

Define como o sistema deve reagir a entradas específicas, o sistema deve se comportar em situações específicas e o que sistema não deve fazer.  Devem ser completos, isso é, atendem a todos os serviços exigidos pelo usuário, e consistentes, isso é, os requisitos não devem ser contraditórios entre si.
